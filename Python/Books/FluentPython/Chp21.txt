# Chapter 21 – Concurrency with Futures (Summary)

## 1. Purpose of the Chapter

This chapter introduces **high-level concurrency using the `concurrent.futures` module**.

The library simplifies the pattern:

> Run many tasks concurrently and collect their results. 

It provides an abstraction over:

* **threads**
* **processes**

and uses the concept of **Futures** to represent asynchronous results.

---

# 2. What Is a Future?

A **Future** represents a computation that:

* may **not be finished yet**
* will eventually **produce a result or exception**

Example idea:

```python
future = executor.submit(func, arg)
```

Here:

* the function runs **asynchronously**
* `future` is an object representing the pending result.

Important properties:

* check if finished
* retrieve result
* retrieve exception. 

---

# 3. Executors

Executors are responsible for **running tasks asynchronously**.

Main types:

| Executor              | Purpose                    |
| --------------------- | -------------------------- |
| `ThreadPoolExecutor`  | concurrency with threads   |
| `ProcessPoolExecutor` | parallelism with processes |

Both share the same API.

Example:

```python
from concurrent.futures import ThreadPoolExecutor

with ThreadPoolExecutor() as executor:
    future = executor.submit(task)
```

Executors manage:

* worker threads/processes
* task scheduling
* result retrieval.

---

# 4. Submitting Tasks

The main method:

```python
executor.submit(function, *args)
```

This:

1. schedules the function
2. returns a **Future object**.

Example:

```python
future = executor.submit(download_flag, cc)
```

Later you retrieve the result:

```python
result = future.result()
```

If the function raised an exception, `.result()` **re-raises it**. 

---

# 5. `executor.map()`

Another way to run many tasks:

```python
executor.map(function, iterable)
```

Example:

```python
executor.map(download_flag, country_codes)
```

Characteristics:

* simpler
* returns results in **input order**
* best when calling **same function with different arguments**. 

---

# 6. `futures.as_completed()`

A more flexible approach.

Instead of returning results in order, it yields **futures as soon as they finish**.

Example:

```python
from concurrent.futures import as_completed

for future in as_completed(futures):
    result = future.result()
```

Advantages:

* handle results immediately
* better for **progress monitoring**
* easier error handling. 

---

# 7. Mapping Futures to Tasks

Because results arrive **out of order**, it’s common to track which task each future represents.

Typical pattern:

```python
future_to_task = {
    executor.submit(task, arg): arg
    for arg in args
}
```

Then:

```python
for future in as_completed(future_to_task):
    arg = future_to_task[future]
```

This allows you to **associate results or errors with the original input**. 

---

# 8. Example Case Study: Downloading Web Files

The chapter uses an example: **downloading country flag images from HTTP servers**.

Three implementations are compared:

1. **Sequential**
2. **ThreadPoolExecutor**
3. **Async version (later in Chapter 22)**

Results:

* sequential → slow
* thread-based concurrency → **dramatically faster** for network tasks. 

Reason:

* downloads are **I/O-bound**
* threads allow waiting operations to overlap.

---

# 9. Progress Tracking with `tqdm`

To visualize progress, the chapter uses:

```python
tqdm(iterable)
```

`tqdm`:

* wraps an iterable
* displays **progress bars**
* estimates remaining time.

It works well with:

```python
as_completed()
```

because tasks finish at unpredictable times. 

---

# 10. Error Handling with Futures

Errors inside concurrent tasks **do not crash immediately**.

Instead:

* exceptions are stored in the **Future**
* raised later when calling `.result()`.

Example:

```python
try:
    result = future.result()
except Exception as exc:
    print("Task failed:", exc)
```

This allows programs to **handle errors individually for each task**.

---

# 11. Thread Pool Size (`max_workers`)

Thread pools limit concurrency.

Example:

```python
ThreadPoolExecutor(max_workers=10)
```

Default value (Python ≥3.8):

```
min(32, os.cpu_count() + 4)
```

Reason:

* enough threads for I/O tasks
* avoid creating too many threads. 

---

# 12. Process Pools for CPU Work

For CPU-bound tasks, use:

```python
ProcessPoolExecutor
```

Benefits:

* bypasses the **GIL**
* uses multiple CPU cores.

Example application in the chapter:

* **prime number checking**

Each process runs independently on a different core.

---

# 13. When to Use Futures

`concurrent.futures` is best for problems where tasks are:

* **independent**
* easily parallelizable
* “**embarrassingly parallel**”.

Examples:

* downloading files
* image processing
* batch data processing
* running independent calculations.

---

# 14. Futures vs Async Programming

Futures are the **foundation for async programming**.

They are used internally in:

* `asyncio`
* event loops
* async tasks.

However:

| Futures              | Async            |
| -------------------- | ---------------- |
| thread/process based | event-loop based |
| simpler mental model | more scalable    |

Async programming is covered in **Chapter 22**.

---

# 15. Key Takeaways

### 1️⃣ `concurrent.futures` simplifies concurrency

You can add concurrency to sequential code with minimal changes.

---

### 2️⃣ Futures represent pending results

They encapsulate:

* result
* exception
* execution state.

---

### 3️⃣ Executors manage task execution

Two main types:

* `ThreadPoolExecutor`
* `ProcessPoolExecutor`.

---

### 4️⃣ `submit()` schedules tasks

Returns a Future object.

---

### 5️⃣ `as_completed()` handles results as soon as they finish

Useful for progress reporting and error handling.

---

### 6️⃣ Thread pools are great for I/O tasks

Example:

* web requests
* file operations.

---

### 7️⃣ Process pools are best for CPU-bound work

They avoid the **GIL** and use multiple cores.

---

**Core message of Chapter 21**

> The `concurrent.futures` module provides a simple and powerful abstraction for running tasks concurrently using threads or processes, with Futures acting as placeholders for asynchronous results.
