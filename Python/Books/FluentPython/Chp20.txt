# Chapter 20 — Concurrency Models in Python (Summary)

## 1. Concurrency vs Parallelism

The chapter starts by clarifying two commonly confused concepts.

### Concurrency

* **Handling multiple tasks at the same time conceptually**
* Tasks make progress **interleaved**
* Works even on **single-core CPUs**

### Parallelism

* **Executing multiple computations simultaneously**
* Requires **multiple CPU cores or processors**

Key idea:

> Parallelism is a special case of concurrency. 

---

# 2. Core Concurrency Terminology

Important terms used throughout the chapter:

### Process

* A **running program instance**
* Has its **own memory space**
* Communicates with other processes via pipes, sockets, etc.

### Thread

* An **execution path inside a process**
* Threads **share memory**
* Easier communication but risk of **data corruption**

### Locks

* Mechanisms used by threads to **avoid race conditions**

### Contention

* Multiple tasks competing for the same resource (CPU, lock, etc.). 

---

# 3. Python’s Three Main Concurrency Models

Python provides **three primary approaches**:

| Model      | Library           | Best for                |
| ---------- | ----------------- | ----------------------- |
| Threads    | `threading`       | I/O-bound tasks         |
| Processes  | `multiprocessing` | CPU-bound tasks         |
| Coroutines | `asyncio`         | Massive I/O concurrency |

These are illustrated with a simple example: a **spinner animation while a slow task runs**. 

---

# 4. Threads (`threading`)

Threads run **within the same process**.

Example idea:

* main thread runs a slow computation
* secondary thread runs a spinner animation

Important properties:

* **share memory**
* easy communication
* require **synchronization** (locks/events)

Example coordination tool:

```python
threading.Event
```

Used to signal when a thread should stop. 

---

# 5. The Global Interpreter Lock (GIL)

The **GIL is the most controversial part of Python concurrency**.

### What it does

* Allows **only one Python thread to execute at a time**
* Protects interpreter internals like reference counting.

Implication:

* **Threads cannot fully use multiple CPU cores for CPU-bound work**

However:

* Python periodically releases the GIL (≈ every 5ms) to allow thread switching. 

---

# 6. Multiprocessing (`multiprocessing`)

Multiprocessing solves the GIL limitation.

Instead of threads:

* **multiple Python processes are created**

Each process:

* has **its own Python interpreter**
* has **its own GIL**

Therefore:

* programs can **use multiple CPU cores**

Example concept:

```python
multiprocessing.Process
```

Behavior:

* similar API to `threading`
* but processes **do not share memory**.

This means:

* data must be **serialized and sent between processes**, which adds overhead. 

---

# 7. Async Concurrency (`asyncio`)

Async programming uses **coroutines and an event loop**.

Instead of threads:

* a **single thread** runs many tasks cooperatively.

Key components:

### Event loop

* schedules tasks
* waits for I/O events
* resumes tasks when ready.

### Tasks

Created using:

```python
asyncio.create_task()
```

Example:

```python
async def supervisor():
    spinner = asyncio.create_task(spin())
    result = await slow()
```

Important property:

* **all coroutines run in a single thread**.

Therefore:

> CPU-bound work blocks the entire event loop. 

---

# 8. Comparing the Three Models

| Feature         | Threads        | Processes | Async             |
| --------------- | -------------- | --------- | ----------------- |
| Memory          | Shared         | Separate  | Shared            |
| CPU parallelism | Limited by GIL | Yes       | No                |
| Best for        | I/O tasks      | CPU tasks | Massive I/O tasks |
| Complexity      | Medium         | High      | Medium            |

Key rule:

* **CPU-bound → multiprocessing**
* **I/O-bound → threads or asyncio**

---

# 9. Experiment: Prime Number Computation

The chapter compares performance using a **prime number computation**.

Results:

### Async version

* **bad for CPU work**
* blocks event loop.

### Thread version

* works but **limited by GIL**.

### Multiprocessing version

* **best performance**
* uses multiple cores.

Conclusion:

> Only processes allow Python to fully use multicore CPUs. 

---

# 10. Communication Between Processes

Processes cannot share memory.

Instead they use:

* queues
* pipes
* message passing

Example pattern:

```
Producer → Queue → Worker processes → Result queue
```

This is the basis for **task queue systems**.

---

# 11. Real-World Python Concurrency Systems

Large Python systems use higher-level tools:

### WSGI application servers

Examples:

* Gunicorn
* uWSGI

Used for **scalable web servers**.

### Distributed task queues

Examples:

* Celery
* RQ

Architecture:

```
Web app → message queue → worker processes
```

Workers process jobs asynchronously and return results. 

---

# 12. Big Picture: When the GIL Matters

The chapter stresses that **the GIL is often overblamed**.

Many workloads are not affected:

* system administration scripts
* network services
* web servers
* data pipelines

Real-world systems work around it using:

* multiprocessing
* distributed systems
* external services. 

---

# 13. Key Takeaways

### 1️⃣ Concurrency ≠ Parallelism

Concurrency structures programs to handle many tasks.

---

### 2️⃣ Python has three main concurrency tools

* `threading`
* `multiprocessing`
* `asyncio`

---

### 3️⃣ GIL limits thread-based parallelism

Threads are good for **I/O-bound tasks**, not CPU-heavy work.

---

### 4️⃣ Multiprocessing enables real parallelism

Because each process has its own interpreter and GIL.

---

### 5️⃣ Async programming scales best for I/O

But **CPU work blocks the event loop**.

---

### 6️⃣ Real systems combine many approaches

Examples:

* web servers
* worker queues
* distributed systems.

---

**Core idea of Chapter 20**

> Python supports concurrency through threads, processes, and asynchronous coroutines—each suited to different workloads—and understanding their trade-offs is essential for building scalable systems.
