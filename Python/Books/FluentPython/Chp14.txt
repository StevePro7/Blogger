# Chapter 14 – Inheritance: For Good or For Worse (Summary)

## 1. Main Idea of the Chapter

The chapter discusses **when inheritance is useful and when it causes problems**.

The key message:

> Inheritance is powerful but often misused; **composition and mixins are usually better choices**.

Python supports complex inheritance patterns (like multiple inheritance), but they must be used **carefully**.

---

# 2. Subclassing Built-in Types Is Tricky

Subclassing Python’s built-in types like:

```python
list
dict
str
set
```

can lead to **unexpected behavior**.

Example problem:

Some built-in methods **bypass overridden methods** in subclasses.

Example:

```python
class MyDict(dict):
    def __setitem__(self, key, value):
        print("Setting item")
        super().__setitem__(key, value)
```

Certain dictionary operations may **not call your overridden method**, because built-in implementations call internal C-level functions.

### Recommended solution

Use **collections.UserDict**, **UserList**, or **UserString** instead.

Example:

```python
from collections import UserDict
```

These classes are designed to be **safe for subclassing**.

---

# 3. Multiple Inheritance

Python allows a class to inherit from **multiple parents**:

```python
class Child(A, B):
```

This is powerful but complicated because Python must decide:

> Which method should be used if multiple parents define it?

This is solved using **Method Resolution Order (MRO)**.

---

# 4. Method Resolution Order (MRO)

The **MRO** determines the order Python searches for methods.

You can inspect it with:

```python
ClassName.__mro__
```

or

```python
ClassName.mro()
```

Python uses the **C3 linearization algorithm** to ensure:

* consistent order
* predictable inheritance behavior

Example search order:

```
Child → Parent1 → Parent2 → ... → object
```

The MRO ensures each class appears **only once in the hierarchy**.

---

# 5. Real-World Multiple Inheritance

Multiple inheritance is widely used in **Python frameworks**.

Examples include:

* GUI libraries
* web frameworks
* mixin-based architectures

A typical pattern:

```
Concrete class
    ↓
Mixin classes
    ↓
Base framework classes
```

This structure allows **reusable functionality without duplication**.

---

# 6. Mixins

A **mixin** is a class designed **only to add functionality** to other classes.

Characteristics of mixins:

* Not meant to be instantiated alone
* Provide small, reusable behavior
* Usually rely on other classes for core functionality

Example:

```python
class LoggingMixin:
    def log(self, msg):
        print(msg)
```

Used as:

```python
class MyClass(BaseClass, LoggingMixin):
```

Mixins allow **code reuse without deep inheritance hierarchies**.

---

# 7. Guidelines for Multiple Inheritance

The chapter provides **important design rules**.

### 1. Distinguish Interface vs Implementation Inheritance

Two kinds of inheritance:

| Type                       | Purpose               |
| -------------------------- | --------------------- |
| Interface inheritance      | define API / protocol |
| Implementation inheritance | reuse code            |

Example:

* **ABCs** → interface inheritance
* **mixins** → implementation inheritance

---

### 2. Use ABCs to Define Interfaces

Abstract Base Classes help clarify **what methods subclasses must implement**.

Example:

```python
from abc import ABC
```

Benefits:

* explicit design
* better documentation
* safer subclassing

---

### 3. Use Mixins for Code Reuse

Instead of deep inheritance trees, prefer:

```
Base class + small mixins
```

This makes behavior **modular and reusable**.

---

### 4. Make Mixins Explicit by Naming

Mixin classes should be clearly labeled.

Example:

```
JSONMixin
LoggingMixin
SortableMixin
```

This tells developers the class is **not meant to stand alone**.

---

### 5. An ABC May Also Be a Mixin

An **Abstract Base Class** can sometimes also provide reusable code.

Example:

```
ABC + partial implementation
```

But:

> A mixin should not usually act as a full interface.

---

### 6. Avoid Multiple Concrete Base Classes

Bad design example:

```
class Child(Database, Network, FileSystem):
```

This causes **conflicting implementations**.

Rule:

> A class should inherit from **only one concrete class**.

Other parents should be mixins.

---

### 7. Provide Aggregate Classes

Framework designers should provide **ready-to-use classes** that combine mixins.

Example structure:

```
Base class
Mixin1
Mixin2
↓
Prebuilt class combining them
```

This simplifies usage for developers.

---

### 8. Favor Composition Over Inheritance

This is the **most important rule**.

Instead of:

```
class Car(VehicleEngine)
```

Prefer:

```
class Car:
    engine = Engine()
```

Composition gives:

* better modularity
* easier maintenance
* fewer inheritance problems

---

# 8. Example: Tkinter

The chapter analyzes **Tkinter**, Python’s GUI library.

Tkinter heavily uses **multiple inheritance and mixins**, which makes its hierarchy:

* powerful
* but difficult to understand

It demonstrates both:

* the **power**
* and the **complexity**

of Python’s inheritance model.

---

# 9. Example: Django Mixins

A modern example appears in **Django generic views**.

Django uses **many mixins** to add features like:

* authentication
* permissions
* form handling
* pagination

Example structure:

```
View
  + LoginRequiredMixin
  + PermissionRequiredMixin
```

This demonstrates how mixins enable **flexible, modular frameworks**.

---

# 10. Key Design Principles (Most Important)

### 1. Use inheritance for **is-a relationships**

Example:

```
Dog is an Animal
```

---

### 2. Use composition for **has-a relationships**

Example:

```
Car has an Engine
```

---

### 3. Use mixins for **shared behavior**

Example:

```
class SerializableMixin
```

---

### 4. Avoid deep inheritance hierarchies

They become difficult to maintain.

---

# 11. Key Takeaways

### Subclassing built-ins can be unsafe

Use `UserDict`, `UserList`, etc.

### Multiple inheritance is powerful but complex

Use **MRO** to resolve method conflicts.

### Mixins are the preferred way to reuse behavior

They keep classes small and modular.

### ABCs define interfaces

They clarify expected behavior.

### Composition is usually better than inheritance

This is the most important guideline.

---

**Core message of Chapter 14**

> Inheritance should be used carefully. Prefer **composition and mixins**, and use inheritance mainly to define clear type relationships.
