# Chapter 18 – Context Managers and `else` Blocks (Summary)

## 1. Main Topics of the Chapter

This chapter explains three important Python control-flow features:

1. **`else` clauses in `for`, `while`, and `try`**
2. **Context managers and the `with` statement**
3. **Utilities from the `contextlib` module**

The central idea is that **`with` blocks help manage setup and teardown operations safely and cleanly**. 

---

# 2. The `else` Clause Beyond `if`

Most programmers know `else` with `if`, but Python also allows it in:

* `for`
* `while`
* `try`

### `for/while ... else`

The `else` block runs **only if the loop finishes normally (no `break`)**.

Example idea:

```python
for item in items:
    if item == target:
        break
else:
    raise ValueError("target not found")
```

Meaning:

* If the loop **breaks**, `else` is skipped.
* If the loop **completes**, `else` runs. 

---

### `try ... except ... else`

The `else` block runs **only if no exception occurs**.

Better structure:

```python
try:
    dangerous_call()
except OSError:
    log_error()
else:
    after_call()
```

Advantages:

* Keeps the **try block minimal**
* Makes it clear **which code may raise exceptions**. 

---

# 3. EAFP vs LBYL

This chapter reinforces an important Python philosophy.

### EAFP

**Easier to Ask Forgiveness than Permission**

```python
try:
    value = mapping[key]
except KeyError:
    ...
```

### LBYL

**Look Before You Leap**

```python
if key in mapping:
    value = mapping[key]
```

Python often favors **EAFP** because:

* fewer condition checks
* avoids race conditions in concurrent programs. 

---

# 4. Context Managers and the `with` Statement

A **context manager** controls the behavior of a `with` block.

Example:

```python
with open("file.txt") as f:
    data = f.read()
```

Here:

* file is **opened**
* code executes
* file is **automatically closed**

The `with` statement replaces the common pattern:

```python
try:
    ...
finally:
    cleanup()
```

It ensures cleanup **even if exceptions occur**. 

---

# 5. Context Manager Protocol

A context manager implements two special methods:

### `__enter__()`

Called when entering the `with` block.

Returns the object bound to `as`.

### `__exit__(exc_type, exc_value, traceback)`

Called when leaving the `with` block.

Arguments provide information about exceptions.

Example structure:

```python
class MyContext:

    def __enter__(self):
        setup()
        return resource

    def __exit__(self, exc_type, exc_value, traceback):
        cleanup()
```

The `__exit__` method can **suppress exceptions** by returning `True`. 

---

# 6. How `with` Actually Works

Execution order:

1️⃣ Evaluate expression after `with` → create context manager
2️⃣ Call `__enter__()`
3️⃣ Execute block
4️⃣ Call `__exit__()` when leaving block

Important detail:

* the **object returned by `__enter__`** is what gets assigned in `as`.
* not necessarily the context manager itself. 

---

# 7. Custom Context Managers

Example pattern:

```python
class LookingGlass:

    def __enter__(self):
        modify_environment()
        return something

    def __exit__(self, exc_type, exc_value, traceback):
        restore_environment()
```

Use cases:

* temporary environment changes
* locking resources
* transactions
* logging or timing blocks.

# 8. `contextlib` Utilities

Before writing your own context manager, check the **`contextlib` module**.

Useful tools include:

### `closing(obj)`

Wraps objects with a `close()` method.

### `suppress(ExceptionType)`

Ignore specific exceptions.

```python
with suppress(FileNotFoundError):
    os.remove("temp.txt")
```

### `nullcontext`

A no-op context manager.

Useful when a context manager may or may not be needed.

### `redirect_stdout` / `redirect_stderr`

Temporarily redirect output. 

---

# 9. `@contextmanager` Decorator

Instead of writing a class, you can build a context manager **with a generator**.

Example:

```python
from contextlib import contextmanager

@contextmanager
def my_context():
    setup()
    yield resource
    cleanup()
```

Execution:

* code **before `yield`** → runs at start (`__enter__`)
* code **after `yield`** → runs at exit (`__exit__`) 

This dramatically **reduces boilerplate code**.

---

# 10. `ExitStack`

`ExitStack` handles **dynamic numbers of context managers**.

Example scenario:

* open an unknown number of files

```python
with ExitStack() as stack:
    files = [stack.enter_context(open(f)) for f in filenames]
```

When the block exits:

* `__exit__` is called **in reverse order** (LIFO).
