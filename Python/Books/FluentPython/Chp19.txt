# Chapter 19 – Classic Coroutines (Summary)

## 1. What a Coroutine Is

A **classic coroutine** is a **generator function used as a data consumer instead of a data producer**.

Normal generators:

* **produce values** with `yield`

Coroutines:

* **receive values** via `.send()`.

Example structure:

```python
def simple_coroutine():
    x = yield
    print(x)
```

Usage:

```python
coro = simple_coroutine()
next(coro)      # start coroutine
coro.send(42)   # send value
```

Here:

* `yield` pauses execution
* `.send()` resumes execution and passes a value. 

---

# 2. Coroutines vs Generators

| Generators    | Coroutines     |
| ------------- | -------------- |
| Produce data  | Consume data   |
| Pull model    | Push model     |
| `next()` used | `.send()` used |

Conceptually:

* **Generators = pull values**
* **Coroutines = push values** 

---

# 3. Evolution of Coroutines in Python

Coroutines evolved through several Python enhancements.

### PEP 342 (Python 2.5)

Added:

* `.send(value)`
* `.throw(exception)`
* `.close()`

These allow external control of generators.

### PEP 380 (Python 3.3)

Added:

* `yield from`
* ability to **return values from generators**.

These features made coroutines much more powerful. 

---

# 4. Coroutine States

A coroutine can be in four states:

| State           | Meaning                 |
| --------------- | ----------------------- |
| `GEN_CREATED`   | created but not started |
| `GEN_RUNNING`   | currently executing     |
| `GEN_SUSPENDED` | paused at `yield`       |
| `GEN_CLOSED`    | finished execution      |

These states can be inspected using:

```python
inspect.getgeneratorstate(coro)
```



---

# 5. Priming a Coroutine

A coroutine must reach its **first `yield` before receiving data**.

Therefore you must call:

```python
next(coro)
```

before sending values.

This is called **priming**.

Example:

```python
coro = averager()
next(coro)
coro.send(10)
```

Without priming:

```
TypeError: can't send non-None value to a just-started generator
```

---

# 6. Priming Decorators

To avoid manually priming coroutines, a decorator can do it automatically.

Example pattern:

```python
def coroutine(func):
    def primer(*args, **kwargs):
        gen = func(*args, **kwargs)
        next(gen)
        return gen
    return primer
```

Usage:

```python
@coroutine
def averager():
    ...
```

Now the coroutine is **automatically primed when created**. 

---

# 7. Example: Running Average Coroutine

A classic example is a **running average calculator**.

```python
def averager():
    total = 0
    count = 0
    average = None
    while True:
        term = yield average
        total += term
        count += 1
        average = total / count
```

Usage:

```python
coro = averager()
next(coro)

coro.send(10) → 10
coro.send(30) → 20
coro.send(5)  → 15
```

Advantages:

* local variables store state
* no classes or closures needed. 

---

# 8. Coroutine Termination

Coroutines may run **infinite loops** and must be terminated intentionally.

Ways to terminate:

### 1. `.close()`

```python
coro.close()
```

Raises `GeneratorExit` inside coroutine.

---

### 2. Sentinel values

Example:

```python
coro.send(None)
```

Used to signal termination.

---

### 3. Exceptions

Coroutines support:

```python
coro.throw(ExceptionType)
```

This injects an exception inside the coroutine. 

---

# 9. Exception Handling in Coroutines

Exceptions can be handled inside the coroutine:

```python
try:
    x = yield
except DemoException:
    handle_error()
```

If an exception is **not handled**, the coroutine **terminates**. 

---

# 10. Returning Values from Coroutines

Since Python 3.3, generators can return values.

Example:

```python
def averager():
    ...
    return Result(count, average)
```

When the coroutine ends:

```
StopIteration(result)
```

The returned value is stored in:

```
StopIteration.value
```

This mechanism allows coroutines to **produce final results**. 

---

# 11. `yield from`

`yield from` simplifies delegation to another generator.

Example:

```python
yield from subgenerator()
```

Key components:

1. **Delegating generator** (uses `yield from`)
2. **Subgenerator**
3. **Caller/client**

`yield from`:

* passes values
* passes exceptions
* returns final results automatically. 

---

# 12. Coroutine Pipelines

Coroutines can be connected into **data-processing pipelines**.

Example structure:

```
data source → coroutine → coroutine → coroutine
```

Each coroutine:

* receives data
* processes it
* forwards results.

This is similar to **Unix pipelines**.

---

# 13. Coroutines for Concurrency

Coroutines can simulate **concurrent tasks**.

Example:

**Taxi fleet simulation**

Idea:

* each taxi is a coroutine
* event loop schedules them
* tasks yield control voluntarily.

This is called **cooperative multitasking**.

Comparison:

| Coroutines        | Threads          |
| ----------------- | ---------------- |
| cooperative       | preemptive       |
| explicit yielding | OS scheduler     |
| single thread     | multiple threads |

Coroutines yield control so other tasks can run. 

---

# 14. Relationship to Async Programming

Classic coroutines are the **foundation of modern async programming**.

Evolution:

```
Generators → Classic coroutines → Native coroutines (async/await)
```

Modern Python async features build on these ideas.

---

# 15. Key Takeaways

### 1️⃣ Coroutines are generators used as data consumers

They receive values via:

```
.send()
```

---

### 2️⃣ Coroutines maintain state naturally

Local variables persist between suspensions.

---

### 3️⃣ Coroutines require priming

Must reach the first `yield` before receiving data.

---

### 4️⃣ `yield from` simplifies coroutine composition

Allows delegation and value passing.

---

### 5️⃣ Coroutines enable cooperative multitasking

Tasks voluntarily yield control.

---

### 6️⃣ Classic coroutines are foundational

They led to:

```
async / await
asyncio
```

in modern Python.

**Core message of Chapter 19**

> Coroutines extend generators to support bidirectional communication and cooperative multitasking, forming the conceptual foundation of Python’s asynchronous programming model.
