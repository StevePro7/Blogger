# Chapter 16 – Operator Overloading: Doing It Right (Summary)

## 1. Main Idea of the Chapter

Operator overloading allows **user-defined classes to work with Python operators** like:

```
+   -   *   @   ==   <   >=
```

by implementing **special methods (dunder methods)**.

Example:

```python
v1 + v2
```

calls:

```python
v1.__add__(v2)
```

The chapter shows **how to implement operators correctly and safely**.

Key idea:

> Operator overloading should make code **more readable and intuitive**, not confusing. 

---

# 2. Restrictions of Operator Overloading in Python

Python intentionally limits operator overloading to prevent abuse.

Restrictions:

1. You **cannot overload operators for built-in types**.
2. You **cannot create new operators**.
3. Some operators **cannot be overloaded**:

   * `is`
   * `and`
   * `or`
   * `not` 

However, many operators **can be overloaded** using special methods.

---

# 3. Categories of Operators

Python operators correspond to **special methods**.

### Unary operators

| Operator | Method    |
| -------- | --------- |
| `-x`     | `__neg__` |
| `+x`     | `__pos__` |
| `abs(x)` | `__abs__` |

Example:

```python
def __neg__(self):
    return Vector(-x for x in self)
```

---

### Arithmetic (Infix) Operators

| Operator | Method        |
| -------- | ------------- |
| `+`      | `__add__`     |
| `-`      | `__sub__`     |
| `*`      | `__mul__`     |
| `/`      | `__truediv__` |
| `@`      | `__matmul__`  |

Example:

```python
def __add__(self, other):
    return Vector(a + b for a, b in zip(self, other))
```

These operators typically **create new objects rather than modifying operands**. 

---

# 4. Unary Operators

Unary operators work on **one operand**.

Examples:

```
-x
+x
abs(x)
```

Typical behavior:

* return a **new object**
* do **not modify the original object**

Example idea:

```python
def __neg__(self):
    return Vector(-x for x in self)
```

---

# 5. Overloading `+` (Vector Addition)

Example behavior:

```python
v1 + v2
```

Vector addition combines elements.

Example:

```
Vector([3,4,5]) + Vector([6,7,8])
= Vector([9,11,13])
```

The implementation often uses `zip_longest` to support vectors of different lengths. 

Important rule:

> Infix operators should **create new objects and not modify operands**. 

---

# 6. Supporting Mixed Types

Your class should work with **different operand types when possible**.

Example:

```
Vector + tuple
Vector + Vector2d
```

To support this, the operator method should return:

```
NotImplemented
```

when it cannot handle the operand.

Example:

```python
def __add__(self, other):
    try:
        ...
    except TypeError:
        return NotImplemented
```

Returning `NotImplemented` allows Python to try the **reverse operator**.

---

# 7. Reverse Operators (`__radd__`, etc.)

When evaluating:

```
a + b
```

Python follows this process:

1. Call `a.__add__(b)`
2. If it returns `NotImplemented`, call `b.__radd__(a)`
3. If both fail → raise `TypeError` 

Example:

```python
def __radd__(self, other):
    return self + other
```

Reverse operators allow operations like:

```
10 + vector
```

---

# 8. Overloading `*` (Scalar Multiplication)

Example:

```
Vector([1,2,3]) * 10
```

Result:

```
Vector([10,20,30])
```

Implementation:

```python
def __mul__(self, scalar):
    return Vector(x * scalar for x in self)
```

To support:

```
10 * vector
```

we also implement:

```python
__rmul__
```

---

# 9. Matrix Multiplication Operator `@`

Python introduced `@` for **matrix multiplication**.

Example:

```
A @ B
```

Method:

```
__matmul__
```

Used in:

* linear algebra libraries
* scientific computing
* NumPy-like APIs

---

# 10. Rich Comparison Operators

Comparison operators include:

```
==   !=   <   <=   >   >=
```

Methods:

| Operator | Method   |
| -------- | -------- |
| `==`     | `__eq__` |
| `!=`     | `__ne__` |
| `<`      | `__lt__` |
| `<=`     | `__le__` |
| `>`      | `__gt__` |
| `>=`     | `__ge__` |

Special behavior:

* reverse comparisons may reuse the same methods
* if comparisons fail, Python may fall back to comparing **object IDs** for `==` and `!=` instead of raising errors. 

---

# 11. Augmented Assignment Operators

Operators like:

```
+=  -=  *=  /=
```

have **special methods**.

Example:

```
__iadd__
__imul__
```

Example:

```
a += b
```

Default behavior:

```
a = a + b
```

which creates a **new object**.

But if `__iadd__` is implemented, the object can be **modified in place**. 

Example:

```python
def __iadd__(self, other):
    self.extend(other)
    return self
```

Important rule:

> In-place operators must return `self`.

---

# 12. Duck Typing vs Goose Typing in Operators

When checking operand types:

### Duck typing

Try the operation and catch errors.

Example:

```
try operation
except TypeError
```

Pros:

* flexible
* works with many types

---

### Goose typing

Use explicit type checks:

```
isinstance(obj, SomeABC)
```

Pros:

* more predictable
* clearer error messages

The chapter suggests **libraries often combine both approaches**. 

---

# 13. Important Design Rules

### 1️⃣ Operators should not modify operands

Example:

```
a + b
```

should produce **a new object**. 

---

### 2️⃣ Return `NotImplemented` for unsupported types

This allows Python to try the **reverse operator**.

---

### 3️⃣ Implement reverse operators when needed

Example:

```
__radd__
__rmul__
```

---

### 4️⃣ Use augmented assignment for in-place changes

Example:

```
__iadd__
```

---

# 14. Key Takeaways

### Operator overloading works via special methods

Example:

```
+ → __add__
* → __mul__
```

---

### Python dispatches operators intelligently

```
a.__add__(b)
→ b.__radd__(a)
→ TypeError
```

---

### Infix operators should create new objects

Only augmented assignment operators should modify objects.

---

### Return `NotImplemented`, not exceptions

This enables Python’s operator resolution algorithm.

---

### Operator overloading should improve readability

Poorly designed operators can make code confusing.

---

**Core message of Chapter 16**

> Operator overloading is powerful but should be used carefully to create intuitive, consistent APIs that behave like built-in Python types.
